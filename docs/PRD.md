# PRD · KB-Agent（内部 + Partner 共享 Agent 知识平台）

## 1. 目标用户与核心场景
| 用户角色 | 使用场景 | 期望结果 |
|---------|---------|---------|
| 管理员 | 上传文档、维护分类体系、建空间、授权成员 | 文档被自动归类总结并沉淀；能控制谁能看哪个空间 |
| 内部员工 | 在被授权空间里对话查询知识、获取原文 | 拿到「答案 + 原文链接」，快速定位文件 |
| 外部 Partner | 仅在被授权空间里查询/取件 | 只能访问授权空间内容，看不到其他数据 |
| （未来）任意用户 | 通过 skill（如 SCM）借助知识生成 PANW 配置 | 生成配置方案，人工确认后下发 |

## 3. 明确的 Out of Scope（本次不做）
- 向量检索 / pgvector（用户明确要求先不上，靠原文降低幻觉）
- Skill 插拔框架的完整实现（M4，仅预留抽象与 API 契约）
- SCM skill 端到端下发 PANW 配置（M5，仅预留“确认后执行”审批契约）
- OpenClaw / Codex 引擎接入（预留 Protocol，未来实现）
- 云对象存储（MVP 用本地文件系统，预留 StorageProtocol）
- 文档在线协同编辑、版本历史
- 前端高级特性（SSR 数据预取优化、i18n、主题定制）——MVP 只做四个核心页面

## 4. 已决策（原待确认问题）
| # | 问题 | 决策 |
|---|------|------|
| Q1 | 支持哪些文件类型？ | **所有常见文档格式（Word/PDF/图片/Excel/PPT/md/txt）**，交给 Claude CLI 原生读取，MVP 不自建解析库 |
| Q3 | 对象存储用哪个？ | **MVP 用本地文件系统**（LOCAL_STORAGE_DIR），经 StorageProtocol 抽象，未来换云 |
| Q4 | 用户从哪来？ | **自建用户表 + 邮箱密码**，注册按 **域名后缀白名单**（ALLOWED_EMAIL_DOMAINS）准入 |
| Q5 | 归类同步还是后台？ | **后台任务**，processing_tasks 表记录进度/日志，失败可重试（reprocess） |
| Q1b | 可搜正文从哪来？ | **复用 CLI 产物**：归类时让 CLI 一趟输出 content_text，供 PG 全文检索 |

## 5. 非功能需求
- 性能：面向千级文档；全文检索走 GIN 索引；LLM 调用设超时与并发上限。
- 安全：JWT 鉴权；workspace 强制隔离；密钥走云 secret，禁止硬编码；签名 URL 限时。
- 可维护性：LLM 唯一出口 `engine/`；api 层无业务逻辑；每端点配 pytest（happy + error）。
- 可扩展性：引擎多后端可切换；skill 框架抽象预留；分类体系层级可扩展。
